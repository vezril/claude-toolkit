## What & why
*"An open standard that enables seamless communication and collaboration between AI agents. It provides a common language for agents built using diverse frameworks and by different vendors, fostering interoperability and breaking down silos."* Without it, multi-agent systems need bespoke point-to-point integrations, and agents get reduced to MCP-style tools. A2A exposes an agent **as an agent** â€” it negotiates directly, keeps its autonomy, and stays **opaque** (no exposure of internal logic/memory/tools).

**Design principles:** Simplicity (reuse HTTP, JSON-RPC, SSE), Enterprise Readiness, Asynchronous (long-running, streaming, push), Modality Independence, Opaque Execution.

## Actors
- **User** â€” human or service defining the goal.
- **A2A Client (Client Agent)** â€” acts for the user, initiates communication.
- **A2A Server (Remote Agent)** â€” an HTTP endpoint implementing A2A; *"operates as an opaque (black-box) system."*

## Agent Card
*"A JSON metadata document describing an agent's identity, capabilities, endpoint, skills, and authentication requirements."* The agent's digital business card for discovery.
- Declares: identity, **service endpoint (`url`)**, A2A capabilities, **`securitySchemes`** (auth), a list of **skills**, optional **Extensions**.
- **Published at `/.well-known/agent-card`** (lifecycle shows `GET /.well-known/agent-card`). âš ď¸Ź Early 2025 drafts used `/.well-known/agent.json` / "AgentCard."

## Communication elements
- **Task** â€” *"a stateful unit of work initiated by an agent, with a unique ID and defined lifecycle."* Has `taskId`. For long-running, trackable, multi-turn work.
- **Message** â€” *"a single turn of communication... containing content and a role ('user' or 'agent')."* Has `messageId`. For immediate exchanges or capability negotiation.
- **Part** â€” *"the fundamental content container used within Messages and Artifacts."* Exactly one of: `text` | `raw` (inline bytes) | `url` (external ref) | `data` (structured JSON); optional `mediaType`, `filename`, `metadata`. Makes A2A modality-independent.
- **Artifact** â€” *"a tangible output generated by an agent during a task."* Has `artifactId`, `name`, one+ Parts; can stream incrementally.

## Task lifecycle
- An incoming message yields either a stateless **Message** (immediate) or a stateful **Task**.
- States: interrupted â€” `input-required`, `auth-required`; terminal â€” `completed`, `canceled`, `rejected`, `failed` (wire form e.g. `TASK_STATE_COMPLETED`).
- **Immutability:** terminal tasks don't restart; a follow-up/refinement is a **new task within the same `contextId`**.
- **`contextId`** â€” server-generated; logically groups related Tasks + standalone Messages (a shared session). **`referenceTaskIds`** links a refinement to a prior task. Parallel tasks under one context are allowed.

- HTTP(S); **JSON-RPC 2.0** is the core payload format (v1.0 also offers gRPC and REST/HTTP+JSON bindings).

## Auth
Standard web security. Requirements declared in the Agent Card `securitySchemes` (OpenID Connect, OAuth, API keys). Credentials passed via **HTTP headers, separate from A2A messages** (e.g. JWT via OIDC). HTTPS throughout.

## A2A vs MCP
*"MCP connects models to data and external resources"* (agentâ†”tool, often stateless); *"A2A standardizes communication among agents developed using diverse frameworks"* (agentâ†”agent, stateful, multi-turn, negotiated). Complementary: an A2A remote agent commonly uses MCP servers internally for its own tools. Wrapping a full agent as a tool is "fundamentally limiting."

## Practical
Discover peers via their Agent Card; exchange Messages for quick turns, open a Task for real work; group a conversation under one `contextId`; stream with SSE or register a push webhook for long jobs; declare/validate auth via `securitySchemes`; keep your agent opaque and your card honest. In [[sdlc-orchestration]], reach for A2A once role agents are independently deployed services rather than co-located skills.
